Under the Hood

How the AI Assistant Works

Every query follows a four-step pipeline designed to balance analytical productivity with federal data governance requirements. No AI output drives case action without human review.

1

Natural Language Parsing and Schema Validation

The query is parsed by GPT-4o against a published data dictionary. Before SQL generation, the model confirms that every referenced field, office, and time period maps to a known schema element. Ambiguous queries are flagged for clarification rather than resolved by inference.

NL query → entity extraction → schema validation → intent classification
2

SQL Generation and Row-Level Security Enforcement

Parameterized SQL is generated from the validated intent and executed against Azure SQL with Row-Level Security policies active. The calling user's Entra ID group membership determines which office partitions are visible. The generated SQL is logged verbatim before execution.

validated intent → parameterized SQL → RLS filter predicate → execution
3

HMAC Signing and 7-Year Audit Retention

Every query, the generated SQL, the raw result set, and the formatted response are concatenated and signed with HMAC-SHA256 using a key stored in Azure Key Vault. The signed record is written to Azure Blob immutable storage and indexed in the audit log before the response is returned to the user.

HMAC-SHA256 sign → immutable Blob write → audit index → response served
4

Human Review Required Before Case Action

AI-generated analytics are advisory. Every response is watermarked "Human review required" and the platform enforces a workflow gate: no AI output can be attached to a case record, exported to an official report, or forwarded to a respondent without a named EEOC staff member acknowledging review.

AI output flagged → reviewer assigned → acknowledgment logged → action permitted
