Technical Reference

System Integration Patterns

How data moves between seven applications — from ARC charge intake through disposition and analytics. API contracts, authentication flows, and the MCP protocol layer.

End-to-End Walkthrough

Case Lifecycle

A discrimination charge moving through every system in sequence, from initial filing to final disposition and analytics aggregation.

1

Charge Filed — ARC

A complainant files a discrimination charge at an EEOC field office. Staff enter the charge into ARC (Agency Record Center), the agency's system of record for all Title VII, ADA, ADEA, EPA, and GINA filings. ARC assigns the charge number and creates the investigation record.

# ARC charge record — authoritative source of truth System: ARC (Agency Record Center) Storage: ARC relational database (EEOC data center) Charge fields: charge_number, basis[], statute[], respondent_id, office_code, date_filed, narrative_text Statutes: TITLE_VII | ADA | ADEA | EPA | GINA Access: No direct API — consumed via ARC Integration API only
2

Data Staged — ARC Integration API

The ARCSyncImporter timer function polls ARC every 15 minutes for new and updated eligible cases. It translates ARC's legacy SOAP/REST interfaces into clean JSON, applies SSRF protections, and writes staged records to Azure Table Storage for downstream consumers.

GET /arc/v1/mediation/eligible-cases?since={watermark}&limit=200 GET /arc/v1/enforcement/cases/{charge_number} Auth: Managed Identity → ARC service account (no stored credentials) Pagination: watermark-based (not offset) — safe for large result sets Bus topic: eeoc-arc-staging / subscription: adr-consumer Timer: every 15 minutes (Azure Functions timer trigger)
3

Classification — OFS Triage

The Triage application receives the staged charge via Service Bus. GPT-4o analyzes the charge narrative, identifies applicable statutes, scores investigation priority (A–D), and recommends a resolution track: mediation, full investigation, or conciliation. Stop sequences prevent legal conclusions from the model; a human reviewer must confirm before any action.

Model: Azure OpenAI GPT-4o (gpt-4o-2024-05-13) Stop sequences: ["Legal Advice:", "Legal Conclusion:"] Temperature: 0.2 (classification task — low variance required) Output: { priority: "A"|"B"|"C"|"D", track: "MEDIATION"|"INVESTIGATION"|"CONCILIATION", statutes: string[], confidence: float } Audit: HMAC-SHA256 signed record → Azure Blob WORM (7 yr) Human review: mandatory before routing decision is committed
4

Routing Decision — Human Reviewer

A program analyst reviews the AI classification in the Triage interface. They may accept the recommendation or override it before committing the routing decision. The confirmed track is published to Service Bus, which triggers the appropriate downstream workflow — ADR for mediation cases, or the investigation queue for others.

POST /triage/v1/cases/{case_id}/routing-decision Body: { track, analyst_id, override_reason? } Event published to: eeoc-routing-decisions (Service Bus topic) Subscriptions: adr-routing-sub | investigation-routing-sub Idempotency: decision_id UUID prevents duplicate processing Audit: analyst_id + timestamp + AI recommendation stored together
5

Mediation — ADR Portal

For mediation-track cases, ADR creates the case record, assigns a mediator, and opens the case for scheduling. The mediator conducts joint and caucus sessions, manages documents, and uses AI to draft settlement agreements. Every AI generation is HMAC-signed and requires human review before any document is shared with parties.

POST /api/v1/cases (creates ADR case from routing event) POST /api/v1/cases/{id}/sessions (schedules sessions) POST /api/v1/cases/{id}/documents (uploads documents) POST /api/v1/cases/{id}/ai/draft-agreement (AI drafting) AI audit: HMAC-SHA256 → Azure Blob immutable tier (7-yr WORM) E-signature: DocuSign and Adobe Acrobat Sign via Azure Function Sessions: Redis-backed (no cookie serialization) Auth: Entra ID + app roles (mediator, supervisor, admin)
6

Resolution or Escalation

If the mediation reaches agreement, the mediator records disposition in ADR and the case is closed. The signed settlement is stored and the disposition event is pushed to UDAP for analytics. If mediation fails, a no-cause finding or full investigation is initiated, and the case may be referred to OGC for litigation.

POST /api/v1/cases/{id}/disposition Body: { outcome: "SETTLED"|"NO_CAUSE"|"INVESTIGATION"|"LITIGATION", settlement_amount?, disposition_date } Settled → closed, metrics pushed to UDAP Investigation → returned to investigation queue Litigation → case packet sent to OGC Trial Tool Event bus: eeoc-dispositions topic → udap-consumer
7

Trial Preparation — OGC Trial Tool

When a case proceeds to litigation, the full enforcement file is transferred to the Trial Tool. Attorneys use RAG-powered document search across the complete case file, construct timelines, manage witnesses, and generate research memos. AI-assisted legal research runs with the same stop sequences and human-review requirement as all other AI outputs on the platform.

Case intake: POST /ogc/v1/cases (triggered by litigation disposition event) Document index: Azure AI Search — vector + keyword hybrid Vision: GPT-4o vision for scanned/handwritten documents Stop sequences: ["Legal Advice:", "Legal Conclusion:"] Auth: Entra ID + app roles (attorney, paralegal, supervisor) Audit: all AI calls HMAC-SHA256 signed → WORM blob (7 yr)
8

Analytics Aggregation — UDAP

Outcome data from every stage flows into UDAP (Unified Data Analytics Platform) via the data-middleware ETL pipeline. Leadership sees settlement rates, time-to-resolution, office performance, and charge basis trends through Superset dashboards, a natural language query interface, and JupyterHub notebooks. Metrics are queryable within minutes of disposition events.

ETL: data-middleware pulls from ADR, Triage, OGC via REST Schedule: daily batch + event-driven disposition updates Storage: PostgreSQL (operational) + Azure SQL (analytics warehouse) Dashboards: Apache Superset (read-only views for leadership) NL query: ai-assistant → GPT-4o → parameterized SQL (no string concat) Notebooks: JupyterHub (restricted to analysts with data-science role)
Model Context Protocol

MCP Protocol Deep Dive

The platform uses JSON-RPC 2.0 over HTTPS for cross-application AI tool orchestration. Every application exposes an MCP spoke; the Hub aggregates and routes tool calls.

Transport

JSON-RPC 2.0 over HTTPS. Each message carries a numeric id, method string, and typed params. The protocol is stateless at the HTTP layer; session context is carried in the MCP sessionId header.

Tool Catalog

Each spoke exposes a tools/list endpoint returning a typed schema for every operation it supports. The Hub merges catalogs from all spokes and re-exposes them under a unified namespace, refreshing every five minutes.

Authorization

Tool calls carry a bearer token scoped to the requesting application. The Hub validates the token against Entra ID before forwarding to the spoke. Spokes re-validate independently — no implicit trust between Hub and spoke.

1. Initialization

The client announces its protocol version and capabilities. The server responds with its own capabilities and a session identifier.

Client Request
{ "jsonrpc": "2.0", "id": 1, "method": "initialize", "params": { "protocolVersion": "2024-11-05", "clientInfo": { "name": "eeoc-mcp-hub", "version": "1.0.0" }, "capabilities": { "tools": {} } } }
Server Response
{ "jsonrpc": "2.0", "id": 1, "result": { "protocolVersion": "2024-11-05", "serverInfo": { "name": "eeoc-adr-mcp-spoke", "version": "2.1.0" }, "capabilities": { "tools": { "listChanged": false } } } }

2. Tool Discovery

The Hub calls tools/list on each spoke to build the merged catalog. Each tool entry includes a typed JSON Schema for its input parameters.

Hub → Spoke
{ "jsonrpc": "2.0", "id": 2, "method": "tools/list", "params": {} }
Spoke Response (truncated)
{ "jsonrpc": "2.0", "id": 2, "result": { "tools": [ { "name": "list_cases", "description": "List ADR mediation cases", "inputSchema": { "type": "object", "properties": { "status": { "type": "string" }, "office_code": { "type": "string" }, "limit": { "type": "integer" } } } } /* ... 9 more tools */ ] } }

3. Tool Invocation

An AI agent calls tools/call with the tool name and typed arguments. The Hub routes the call to the appropriate spoke and returns the result. Example: an analytics agent querying active ADR cases.

Agent → Hub
{ "jsonrpc": "2.0", "id": 7, "method": "tools/call", "params": { "name": "adr__list_cases", "arguments": { "status": "ACTIVE", "office_code": "PHI", "limit": 25 } } }
Hub → Agent
{ "jsonrpc": "2.0", "id": 7, "result": { "content": [ { "type": "text", "text": "{\"cases\": [{\"id\": \"ADR-2026-00412\", \"charge_number\": \"530-2026-00812\", \"status\": \"ACTIVE\", \"mediator_id\": \"u-3a8f\", \"office_code\": \"PHI\"}], \"total\": 18}" } ], "isError": false } }

Authentication

Authentication Flows

Three distinct auth patterns covering every actor on the platform: EEOC staff, external parties, and service-to-service calls.

EEOC Staff

Mediators, attorneys, analysts, supervisors, and admins with @eeoc.gov accounts.

  1. 1 Browser navigates to application URL
  2. 2 App redirects to login.microsoftonline.com (Entra ID)
  3. 3 EEOC SSO / MFA completes — Entra issues id_token + access_token
  4. 4 App reads roles[] claim from token — maps to RBAC permissions
  5. 5 Session stored in Redis — no cookie serialization
roles: mediator | supervisor | admin | analyst | attorney | paralegal

External Parties

Charging parties, respondent representatives, and other non-EEOC users requiring portal access.

  1. 1 User navigates to external portal URL
  2. 2 OIDC Authorization Code + PKCE flow begins — redirect to secure.login.gov
  3. 3 Login.gov identity-proofs the user (IAL1 or IAL2)
  4. 4 App exchanges code for token using private_key_jwt client assertion (no shared secret)
  5. 5 Session stored in Redis — access scoped to user's own cases only
scope: cases:read cases:document-upload — own records only

Service-to-Service

Inter-application calls, Azure Function triggers, Service Bus consumers, and Key Vault access.

  1. 1 Service runs under its Azure Managed Identity — no credentials in code or config
  2. 2 Azure IMDS endpoint issues a short-lived bearer token for the target resource
  3. 3 Token presented in Authorization: Bearer header to target service
  4. 4 Target validates against Entra ID JWKS — confirms oid + appid claims
  5. 5 Secrets fetched from Key Vault via the same Managed Identity — never stored in env vars
credentials: zero — no passwords, keys, or connection strings in code
API Reference

API Reference Summary

Key endpoints and MCP tools across the platform. All REST endpoints return RFC 7807 Problem Details on error. All calls propagate X-Request-ID.

  • GET
    /arc/v1/mediation/eligible-cases
    Returns cases eligible for mediation. Watermark-based pagination via since query param. Protected against SSRF.
  • GET
    /arc/v1/enforcement/cases/{charge_number}
    Returns the full enforcement case record for a given ARC charge number.
  • GET
    /arc/v1/mediation/cases/{charge_number}/status
    Current status of a mediation case as reported by ARC — does not mutate ARC data.
  • POST
    /arc/v1/mediation/cases/{charge_number}/close
    Writes disposition outcome back to ARC after mediation closes. Body: { outcome, settlement_amount?, close_date }.
  • GET
    /arc/v1/health
    Liveness check. Returns 200 OK with ARC connectivity status. Called by platform health monitors every 60 seconds.
  • MCP
    list_cases
    List ADR mediation cases with optional filters. Params: status, office_code, mediator_id, limit, cursor.
  • MCP
    get_case
    Fetch full case details by case ID — parties, sessions, documents, current status, and audit trail summary.
  • MCP
    get_case_chat
    Retrieve chat messages from a case channel. Params: case_id, channel (main / complainant / agency).
  • MCP
    get_case_documents
    Fetch document metadata for a case with download URLs. Does not return file content inline.
  • MCP
    create_case
    Create a new ADR mediation case. Requires charge_number, office_code, and statute list. Returns the new case ID.
  • MCP
    update_case_metadata
    Update charge number, statutes, key dates, or assigned office on an existing case. Partial updates supported.
  • MCP
    post_chat_message
    Post a message to a case chat channel. Params: case_id, channel, text. Stored with sender identity and timestamp.
  • MCP
    add_participant
    Add a participant to a case. Params: case_id, user_id, role. Triggers access provisioning for the participant portal.
  • MCP
    remove_participant
    Remove a participant from a case. Revokes portal access and records the removal in the case audit trail.
  • MCP
    reassign_case
    Reassign a case to a different mediator. Params: case_id, mediator_id, reason. Requires supervisor role.
  • GET
    /api/tools
    Returns the merged tool catalog from all registered spokes. Cached for 5 minutes. Used by AI agents for tool discovery.
  • POST
    /api/tools/refresh
    Forces an immediate re-discovery cycle across all registered spokes. Requires hub:admin scope. Returns updated tool count.
  • GET
    /api/spokes
    Returns the list of registered spoke applications with their URLs, last-seen timestamps, and tool counts.
  • POST
    /api/spokes/register
    Registers a new MCP spoke. Body: { name, url, auth_scope }. Requires hub:admin scope. Triggers immediate discovery.
  • GET
    /api/health
    Liveness and readiness check. Returns per-spoke connectivity status and last-refresh timestamp. Returns 200 even when spokes are unreachable (degraded).
  • POST
    /api/v1/query
    Natural language query endpoint. Body: { question: string, context?: object }. GPT-4o generates parameterized SQL — no string concatenation. Returns data + SQL explanation.
  • POST
    /api/v1/narratives
    Generates a plain-language narrative summary of query results. Used by the analytics assistant for leadership briefings. HMAC-signed audit record on every call.
  • GET
    /api/v1/metrics/{metric_key}
    Returns pre-computed metric values by key. Keys: settlement_rate, avg_resolution_days, charge_volume, basis_breakdown. Params: office_code, period.
  • GET
    /api/v1/dashboards
    Returns available Superset dashboard definitions and embed tokens for the calling user's role. Tokens expire in 30 minutes.
  • POST
    /api/v1/ingest
    Accepts metric payloads from source applications (ADR, Triage, OGC, OCHCO). Authenticated by Managed Identity. Body validated against per-source JSON Schema.
